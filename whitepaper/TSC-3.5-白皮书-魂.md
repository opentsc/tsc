# TSC 3.5 白皮书 ·《魂》

**The Thin-Shell Company 3.5 — The Soul: Laws of a Self-Evolving Organization**

> 版本 v3.5 · 前置 v3.0（自创生·VSM·DAO·递归）
> 定位：这是一部**法**，不是一份说明书。它定义一个 TSC 组织"为什么存在、什么算好、怎么进化"。它要同时被人读懂、被 AI 执行。
> 配套文档：《壳》（OpenTSC v1.0 设计理念与技术实现）——魂的运行容器。

---

## 总纲：魂与壳

v3.0 把 TSC 讲成一个会自我进化的组织体。3.5 把它劈成两半，劈得更清楚：

> **TSC 是魂，OpenTSC 是壳。**
> **魂 = 这套法 + 组织积累的全部判断与记忆。** 它抽象、可移植、决定"为什么"和"什么算好"。
> **壳 = 运行这套法的容器。** 它由模块、agent、技能组成，模块化、可替换、可长到硬件，决定"怎么做"。

为什么要这样劈。因为薄壳公司的全部赌注是"一个人驱动一个大组织"。壳越薄，魂必须越厚、越能脱离任何具体的人和任何具体的软件而存活。一个组织如果它的判断力只存在创始人脑子里、它的记忆只存在某一套软件里，那它的壳一破，魂就跟着死。

3.5 要求：**魂能从一个壳里完整倒出来，注进另一个壳里复活。** 软件可以重写，团队可以换人，模型可以升级——只要魂（创世承诺 + 判断法典 + 事件流 + 校准记忆）还在，这个 TSC 就还是同一个 TSC。这是忒修斯之船问题在组织层面的答案，也是「魂壳」这个名字的全部重量。

```
       魂 (Soul) — 可移植、跨壳存活
   ┌──────────────────────────────────┐
   │  创世层      不可变的存在承诺        │
   │  判断法典    什么算好/可信/值得动    │
   │  事件流      带证据的全部轨迹        │
   │  校准记忆    判断对过错过的账本      │
   └──────────────┬───────────────────┘
                  │ 注入 / 复活
   ┌──────────────▼───────────────────┐
   │  壳 (Shell) — OpenTSC，可替换、可重写 │
   │  内核 · 模块接口 · agent · 技能 · 硬件 │
   └──────────────────────────────────┘
```

# 第一部分：十二法则

法则分四族，每族三条。每条用三句话给出：**人话**（任何人能懂）、**机器话**（AI 能执行）、**根**（来自哪条原理）。这十二条是 TSC 的创世层，不可违背；要改，先改这部法。

## 存在族（Laws of Existence）——TSC 是什么

### 法则一 · 创世不可变

- **人话**：一个 TSC 为什么存在、它死也不肯越过的底线，定下就不能改。改了，它就成了另一个 TSC。
- **机器话**：`genesis` 层字段一次写入、永不修改；任何对 genesis 的变更等价于销毁当前实体并新建一个新 `tsc_id`。日常决策不触碰 genesis；只有当其他决策与 genesis 冲突时，genesis 作为最高仲裁出现。
- **根**：自创生条件四（核心不可改变）· VSM System 5（身份层）· 忒修斯之船。

### 法则二 · 魂壳分离

- **人话**：法和记忆（魂）跟运行它的软件（壳）是两样东西。壳坏了，魂能倒进新壳里活过来。
- **机器话**：`soul = {genesis, judgment_codex, event_stream, calibration_memory}` 与 `shell = {kernel, modules, agents, skills}` 物理分离存储。魂是纯数据契约，不依赖任何具体壳实现；任何合规的壳都能加载任意魂并继续运行。复活 = 用旧魂初始化新壳。
- **根**：自创生（自我再生产）· 魂壳（SoulShell）持久记忆与复活机制。

### 法则三 · 身份靠 ID 不靠名字

- **人话**：每个人、每个组织、每个任务，内部都有一个永不变的编号。名字、马甲、账号只是这个编号底下的标签。
- **机器话**：每个实体有唯一不可变 `id`（如 `p_0f3a` / `o_*` / `op_*` / `tsc_*`）。名字、别名、平台账号是 `id` 下的属性，带 `confirmed | suspected` 状态。文件/路径锚点用 ID，永不用名字。
- **根**：身份连续性由创世层定义，不由任何具体组件定义。

> **持有者责任（贯穿存在族）**：魂持有极端信息不对称——你知道每个 NPC 的底，对方不知道被记录。因此本地存储、静态加密、离线优先、访问控制是地基，不是选项。这一条的强度等同法则，写在这里提醒：**魂越值钱，越要锁好。**

## 感知族（Laws of Perception）——TSC 怎么看世界

### 法则四 · 存轨迹不存快照

- **人话**：情报是流动的。系统记的是"什么时候发生了什么"的流水，当前状态是从流水算出来的，不是一个被反复覆盖的字段。
- **机器话**：事件流 append-only。当前状态是 derived view，非持久字段。修正以新事件表达（旧事件软删除标记保留），永不物理覆盖。
- **根**：自创生条件一（内部状态可观测）· 生成式 agent 的"记忆流"。

### 法则五 · 证据先于判断

- **人话**：先有证据，才有判断。每条情报得说清楚"谁说的、多可信"；每条建议得说清楚"凭哪几条情报、什么逻辑"。
- **机器话**：每条事件强制带 `source` + Admiralty 双轴评级（来源可靠性 A–F × 信息可信度 1–6，分开存）。每条建议/预测强制带 `reasoning_chain` + `due_date`，否则不得落库。敌对/负面判断必须带可复审证据链，不得一次定性污染后续全部解读。
- **根**：数据契约 · 反操控 · 生成式 agent 的"幻觉记忆"失效模式（用证据门堵住）。

### 法则六 · 先草案后入册

- **人话**：自动提取出来的东西先当"草稿"，确认过了才算正式情报。
- **机器话**：collector 自动产出进 `inbox/` 为 candidate，状态 `draft | pending_verification`；进正式事件流需用户确认或显式标注待验。原始素材留在 `raw/`，带 manifest 与 hash。
- **根**：避免回音壁与幻觉污染 · 持有者责任。

## 判断族（Laws of Judgment）——TSC 怎么决定什么算好

### 法则七 · 双法典

- **人话**：一个会进化的组织需要两部法。一部管"什么允许做"（规则），一部管"什么算好、谁算可信、什么值得动用"（判断标准）。后者过去只存在你脑子里，现在必须写进系统。
- **机器话**：`rule_codex`（创世/宪法/操作三层，定义允许与禁止、修改门槛与程序）与 `judgment_codex`（评分维度、信任维度、技能等级定义、可调动度公式）并列为两部宪法，共享三层结构与自迭代机制。系统的评价维度不得内置硬编码，全部从 `judgment_codex` 读取。
- **根**：OpenTSC 实战暴露——"评价标准存在模型权重里，不在系统里，换个模型结论就变"。这是 3.5 相对 3.0 最大的概念增量。

### 法则八 · 现实校准，不靠情绪

- **人话**：系统调整自己，只看"预测准不准、有没有用"，绝不看"让创始人舒不舒服"。一个不敢说逆耳判断的系统是废的。
- **机器话**：区分两类反馈——`useful/accurate`（可用于调整生成）与 `comfortable`（仅记录，禁止用于调整）。每条预测到期强制回填实际结果，命中率按情境/知识来源分解，是唯一合法学习信号。操控源包括创始人自己的偏好。
- **根**：进化博弈 · 进化稳定策略（ESS）· 防御性设计（对抗腐化）。

### 法则九 · 属性从事件推导，不靠手填

- **人话**：你不能直接给某个人打分。你只能记录他做了什么，系统按判断法典自动算出他的属性、技能、状态。这样换谁来读，结论都一样。
- **机器话**：所有角色属性（基础/技能/状态）是 derived 值，每个值带 `provenance`（触发它的 event_ids）+ `confidence` + `reviewed_date` + `decay`。禁止直接写入属性；写入只能通过追加事件，由判断引擎按 `judgment_codex` 推导更新。
- **根**：生成式 agent 的"反思"（从记忆合成更高层推断）· 判断可复现性要求。

## 进化族（Laws of Evolution）——TSC 怎么长大

### 法则十 · 按需生长

- **人话**：默认最小。一个实体先是一个文件，重要了才长成文件夹。一个组织先用最少的 agent，撞到墙了才长出新 agent。不预先堆架构。
- **机器话**：实体默认轻量节点，按维度重要性分裂子文档。agent/技能/模块仅在出现**已证实的能力缺口**时由自创生引擎生成（见法则十一）。编排复杂度跟随负载复杂度，不超前。长期不用的模块进日落流程。
- **根**：自创生（边界按需扩展）· 多 agent 工程教训（40% 多 agent 项目失败于过度架构；单 agent 在 64% 任务上不输多 agent——所以先简单）。

### 法则十一 · 四重自创生

- **人话**：组织能自己长出它需要的东西。需要新规则时提案新规则，需要新能力时找人或造 agent，发现更好的目标时改目标，任务需要时扩边界、完事再收回来。
- **机器话**：四个自创生回路均需满足"可观测→可触发响应→有反馈循环→不碰创世核心"：
  - **规则自创生**：规则被反复违反/引用 → 自动提案修改 rule/judgment codex。
  - **成员（agent）自创生**：检测能力缺口 → 生成 agent 或派招募官找人（见第六部分创世清单）。
  - **目标自创生**：执行中发现更高价值机会 → 转化为目标修正提案，走治理流程。
  - **边界自创生**：按任务动态纳入/收回成员（含分叉与合并协议）。
- **根**：自创生四层次 · agent skill 的自主合成与组合式获取。

### 法则十二 · 递归同构

- **人话**：一个人是一个壳，一个团队是一个壳，一个公司是一个壳，整个生态也是。每一层用的是同一套循环、同一套属性、同一部法。地图可以无限往下钻。
- **机器话**：`person ⊂ team ⊂ org ⊂ ecosystem` 全部是 TSC 实体，递归嵌套。同一套"观察→判断→决策→行动"循环和同一套属性本体（基础/技能/状态）在每一层各跑一遍并可向上聚合。每个 System 1 运营单元本身是一个完整可行系统（VSM 递归性）。
- **根**：VSM 的递归性 · 涌现 · TSC^n。

> **烙印协调（进化族的协作规则）**：组织里的多个 agent 不靠一个中央指挥官互相调用，而靠在共享环境（vault/事件流）里读写标记来协调——一个 agent 留下标记，另一个 agent 读到就行动。这让任何 agent 可随时替换、下线、换模型而循环不断。这是"内核稳定、边缘可变"在协作层面的兑现，也是规模化不出瓶颈的前提。

# 第二部分：职业 · 玩家 · NPC 本体论

3.5 引入一套游戏本体，让"谁是谁、谁能干什么"对人和 AI 都一目了然。

## 三类存在

### 玩家（Player）

- **定义**：持有魂的人。做不可逆决策、握有创世层、是系统最终服务的对象。
- **特征**：可以有一个或多个（联合创始人）。玩家是唯一能改创世层、能拍板目标修正、能确认草案入册、能宣告任务完成/放弃的角色。
- **关键**：玩家不被系统"评分"——系统是玩家认知的延伸，不是玩家的考官。但玩家自己的执行力/抗压等属性可被自己的事件校准（玩家也是世界里的一个实体），用于自我管理，不用于他人调度。

### NPC（Non-Player Character）

NPC = 玩家之外的一切行动者，分两个**根本不同**的子类，系统对两者的关系完全不同：

| | Agent NPC（内生 / 系统可控） | 人类 NPC（外生 / 不可控） |
|---|---|---|
| 是什么 | TSC 实例化并控制的 AI agent | 真实的人（成员、客户、对手、潜在合作者） |
| 系统对它 | 可生成、可配置、可重置、可下线 | 只能**观察、判断、尝试调动**，不能控制 |
| 属性来源 | 确定性配置 + 运行表现 | 从事件流**推断**，带不确定性、需校准 |
| 自由意志 | 无（按 prompt + 技能行动） | 有（会变、会说谎、会掉链子、会背叛） |
| 建模架构 | 记忆-反思-规划（可控版） | 记忆-反思-规划（观察推断版） |

**这个区分是 3.5 的硬核**：人类 NPC 永远是"外部环境"的一部分，系统对他们的全部权力只有"看清"和"调动"，没有"控制"。把人类 NPC 当 agent 一样"指派任务"是范畴错误——你只能记录他们的事件、推断他们的属性、预测他们的行为、计算调动他们的成本。

> 两个子类共用同一套建模架构（记忆流→反思→规划→行动，来自生成式 agent 研究），这让系统可以用同一套契约描述所有 NPC；区别只在属性是"配置的"还是"推断的"。

### 职业（Profession / Class）

- **定义**：一个角色原型。它是一份**契约**：规定"要担任这个职业，必须有哪些技能、达到哪些属性门槛、对应 VSM 哪个功能、读写哪些数据"。
- **特征**：
  - 职业 ≠ 人。一个职业可由 Agent NPC 担任（便宜、可重置），也可由人类 NPC 担任（贵、不可控），也可由玩家亲自担任。
  - 一个 NPC 可同时持有多个职业（流式职业系统，参考 Cyberpunk 的 fluid class）。
  - 职业可进化：满足条件升级、长出子职业（参考 LitRPG 的 class evolution）。
  - 职业是壳里**自创生的最小单位**——一个新 TSC 缺什么职业，系统就生成对应 agent 或派人去填（法则十一·成员自创生）。

# 第三部分：角色属性——最超前的架构

属性是这套游戏的世界模型。3.5 把它架构成下面这套，目标是"最超前"且"可被任何 AI 直接计算"。

## 三层属性

```
角色画像 = 基础属性层 + 技能树层 + 状态层
```

- **基础属性层（innate，慢变，半年级）**：执行力上限、学习速度、抗压能力、可靠性基线、自主性。相对稳定，是"天赋"。对 Agent NPC 是配置值，对人类 NPC 是长期推断值。
- **技能树层（learned，事件驱动，节点制）**：有等级、有前置、靠**事件触发升级**（不是花点数买）。例：「报价重构 Lv.5」前置「行业洞察 Lv.3」+「客户心理 Lv.3」。参考动态技能树——能力随真实事件解锁，不靠主观打分。
- **状态层（state，带倒计时的 buff/debuff）**：临时增益/减益。例：「已读不回·可靠性存疑」debuff，一周观察期，不再犯自动清除，再犯固化为永久。结构化字段，可查询、可比较、可自动筛选。

## 六条超前设计原则

这六条是属性架构"超前"的地方，每一条都让属性比传统数据库字段或传统 RPG 数值更强：

1. **属性是推导值，不是存储值**（法则九）。你改不了属性，只能加事件，系统算出属性。保证客观与可复现。

2. **每个属性是一个带证据的概率声明，不是一个数字**。属性值的完整形态：
   ```yaml
   negotiation:
     value: 0.72            # 当前估计
     confidence: 0.6        # 这个估计有多确定
     provenance: [evt_0a1, evt_0b9, evt_112]   # 凭哪些事件
     reviewed: 2026-06-01   # 上次更新
     decay: 0.03/month      # 不再有新证据就衰减
     source_admiralty: B2   # 证据的平均可信度
   ```
   每个信念都可审计、会过期、会衰减。这是"超前"的第一性原理。

3. **Agent NPC 与人类 NPC 同 schema、异来源**。同一份属性结构，Agent 的由配置+表现填，人类的由观察+推断填。系统用同一套查询接口对待所有角色，但永远知道哪些是"配置的确定性"、哪些是"推断的不确定性"。

4. **属性是分形的——同一套本体在每一层复用**（法则十二）。一个人有执行力，一个团队也有执行力（成员聚合），一个公司也有。查 Stanley Team 的谈判能力 = 聚合其成员谈判属性 × 关系健康度。一套 schema，递归套用到 person/team/org/ecosystem。

5. **关系属性是边，不是节点字段**。信任、可调动度、交情成本是"你与他之间"的属性，挂在关系边上，有方向、有时间、需校准。不是某个人身上的一个数。`advisor.leverage` 算的是 `价值 × 可达性 × 关系健康度 − 调动成本`，全部来自边。

6. **属性可预测，不只可描述**。除了"X 现在的谈判能力"，系统存"X 在情境 C 下的预期表现"——一个由预言家+校准喂养的概率分布。属性同时是一个**对未来行为的生成模型**，不只是一个历史画像。这是属性架构指向的终点：从"记录这个人是谁"走到"预测这个人会怎么做"。

# 第四部分：VSM 必备职业（预设最基础职业）

一个 TSC 要"可行"（VSM 意义上能在复杂环境存活），必须覆盖五个子系统的功能。下面是**创建任何 TSC 都必须就位**的最小职业集。每个职业标注：VSM 归属、默认担任者、可否由 Agent 担任、核心职责。

| 职业 | VSM | 默认担任 | Agent 可担任 | 核心职责 |
|---|---|---|---|---|
| **创世者 / 玩家** | S5 | 人（玩家） | 否 | 持有创世层与判断法典，最高仲裁，确认草案，宣告任务终结 |
| **指挥官** | S3/S5 | 玩家 或 顶配 Agent | 部分 | 战略决策，调用所有底层产出，关键时刻触发 |
| **哨兵** | S2/S4 | Agent（廉价高频） | 是 | 盯异常、盯 deadline、盯模式变化；只读，发现就在 vault 留标记 |
| **摄入官** | S4 | Agent（中等） | 是 | 接收新素材、分类分段、识别参与者，多模态，产出 inbox 草案 |
| **蒸馏师** | S3 | Agent（顶配低频） | 是 | 从原料提取情报，更新属性，建因果边——把原料变情报的核心 |
| **预言家** | S4 | Agent（顶配按需） | 是 | 预测人/项目走向，落库带到期日，回填校准 |
| **简报官** | S3 | Agent（中等定时） | 是 | 生成报告与任务分配，启动时浮出该行动的事，**必含逆耳项** |
| **协调员** | S2 | Agent | 是 | 检测并化解资源冲突与接口冲突，不管理单元只协调接口 |
| **执行者** | S1 | 人 或 Agent | 是 | 直接产出 TSC 核心输出，在职责内自治 |
| **归因官 / 司库** | S3 | Agent + 玩家确认 | 部分 | 贡献归因、资源（算力/注意力）分配、贡献代币发放，看证据不看关系 |
| **招募官 / 猎头** | S4 | Agent | 是 | 检测能力缺口，在能力市场搜寻人才，产出招募意图或提示玩家去找人 |

**说明**：
- 这 11 个是 VSM-完整的最小集。一个一人 TSC 起步时，玩家亲自兼任创世者+指挥官，其余尽量交给廉价 Agent，撞到墙才升级（法则十）。
- 招募官是法则十一"成员自创生"的执行者：当某个必备职业既无合适 Agent 可担任、又无合适人类 NPC 在册，招募官产出"需要一个能做 X 的人"的招募意图，系统**指定 Agent 去公开渠道搜寻，或提示玩家去找人**。
- 不同 TSC 会在这个基础集上**自创生出领域职业**：交易 TSC 长出「风控官」「行情摄入官」；社群 TSC（如 HerName.AI）长出「成员活跃度哨兵」「情绪健康顾问」；内容 TSC 长出「选题预言家」「爆款蒸馏师」。基础集恒定，领域职业按需生长。

# 第五部分：创世清单——创建一个 TSC 必须先告诉系统的事

法则要求：**创建 TSC 时必须确定的事，必须提前告诉系统。** 系统据此算出缺口，再决定"派 Agent 去补"还是"叫玩家去找人"。这是这套游戏的"角色创建界面"。

## 七问（缺一不可，缺则系统拒绝启动）

1. **存在宣言**：这个 TSC 为什么存在？它死也不肯越过的底线是什么？→ 写入创世层（法则一），永不可变。
2. **初始目标**：现在要达成什么？→ 可演化（目标自创生），但必须有起点。
3. **玩家**：谁持有魂？一人还是几人？各自的不可逆权限边界？
4. **环境**：在什么世界里运作（市场 / 社群 / 内容 / 交易 / 政企…）？→ 决定需要哪些 collector / 传感器、长出哪些领域职业。
5. **判断法典种子**：在这个 TSC 里，"好 / 可信 / 值得动用"分别意味着什么？→ 初始评分维度与信任维度（法则七、九）。
6. **资源边界**：模型预算、玩家注意力预算？→ 决定哪些职业用顶配 Agent、哪些用廉价 Agent、哪些必须省着用人类。
7. **预期成员类型**：会和哪些 NPC 打交道（哪类 Agent + 哪类人）？

## 系统在七问之后做的事（自动）

```
读七问
 → 计算 VSM 必备职业是否都有合适担任者（缺口分析）
 → 对每个缺口职业：
     · 能由 Agent 担任？      → 自创生引擎生成该 agent（草案，待玩家批准）
     · 必须人类、且在册有合适人选？ → 提名候选 + 调动成本估计
     · 必须人类、在册无合适人选？   → 派招募官产出招募意图；或直接提示玩家："你需要找一个能做 X 的人"
 → 长出环境对应的领域职业槽（先空着，按需填）
 → 初始化魂：写 genesis、judgment_codex 种子、空 event_stream、空 calibration_memory
 → 启动壳：挂载内核 + 必备模块
```

这一步把"成员机制自创生"从抽象原理变成创建组织时的第一个具体动作。系统从出生第一刻就知道自己缺什么，并主动去补。

# 第六部分：生命周期——健康、死亡、复活

## 健康指标（系统活得好不好）

- **S5 很少被触发** = 判断标准稳定、价值没漂移（日常看不见创世层，是健康的标志）。
- **校准命中率随时间上升** = 魂在真的学习，不是回音壁。
- **判断法典在迭代** = 判断标准在进化，不僵死。
- **简报官能主动浮出逆耳项** = 没退化成讨好玩家的机器。

## 死亡条件（系统失效）

- 校准命中率停止更新，预测不再回填 → 感知与判断脱节，退化成带标签的文件夹。
- 判断法典一年没动过 → 判断力石化。
- 系统只说玩家爱听的 → 被玩家偏好操控，沦为昂贵回音壁（法则八失守）。
- 创世层被随意篡改 → 身份连续性断裂，已是另一个 TSC。

## 复活（法则二的兑现）

壳坏了、要重写、要换模型、要迁移——不影响魂。复活流程：

```
导出魂 = {genesis, judgment_codex, event_stream, calibration_memory}
 → 校验魂的契约完整性（事件流可推导出当前状态？校准账本连续？）
 → 用魂初始化一个全新的壳（可以是完全不同的软件实现）
 → 壳按魂重建：从事件流重算全部属性，从校准记忆恢复判断力，从 judgment_codex 恢复评价标准
 → 同一个 tsc_id 继续运行
```

只要创世层不变、事件流连续、校准账本完整，复活后的 TSC 就是同一个 TSC。这是「魂壳」架构对组织永续的承诺：**身体可以一再重造，魂只要不丢，组织就不死。**
